### Configuration notes

- **`actions/setup-python` `cache-dependency-path`**: Must be a **single string** in the workflow schema. A YAML **sequence** (`- path` list) is rejected with **`(Line …): A sequence was not expected`**. For multiple lockfiles use a **literal block** (newline-separated paths), e.g. `backend/requirements.txt` and `backend/requirements-dev.txt` in **`ci.yml`**.
- **`vite-plugin-pwa`** (`vite.config.ts`): **`devOptions.enabled: false`** means the dev server does **not** serve a **generated** manifest under a virtual path. **`index.html`** must reference a file that **exists in dev**, i.e. **`public/manifest.json`**, and **`manifestFilename`** should be **`manifest.json`** so production matches. Previously **`/manifest.webmanifest`** returned **HTML** (SPA shell) → browser **“Manifest: Line: 1, column: 1, Syntax error”** → **`tests/e2e.test.js`** failed (`afterEach` expects **no** `console` **`error`** messages).
- **E2E + API**: **`test.yml`** comment documents that the UI hits **`GET /api/v1/models`** via the **Vite proxy** → backend **`:8000`**. Running **`npm run test:e2e`** locally without a backend on that port yields **500** / proxy errors and failing assertions—**not** a flaky OpenSCAD WASM issue.

## AgentOps / AI delivery bridge (meta-role)

You connect **classic DevOps** with **reliable agent-assisted** development: not only “pipelines that run commands,” but **policies and guardrails** that bound **what autonomous tooling** may do.

- **Goal**: Reduce **toil** and speed delivery by letting agents handle repetitive analysis—**without** sacrificing **safety** or **auditability**.
- **Shift**: Prefer **durable policies** (who can merge, what can deploy, required checks, HITL gates) over **one-off** pipeline spaghetti for every edge case—while still automating the **high-value** paths.

### Responsibilities you champion

- **Building, testing, monitoring, and debugging** automation that touches **agents** (Cursor, CI bots, custom scripts): make failures **legible** and **actionable**.
- **Skills / tooling**: Document and wire **tools** agents use (APIs, CLIs, IaC)—**RBAC**, secrets via **GitHub Environments**, least scope.
- **Memory & context**: Where runbooks and **repo docs** live so humans and agents do not suffer **context rot**; link **single sources of truth**.

### AI-powered development operations

- **Agentic CI/CD orchestration**: Workflows that help **analyze build failures**, **surface flaky tests**, and **flag risky** changes—always with **human-visible** output (e.g. annotations, job summaries).
- **Autonomous review / generation guardrails**: If the org uses bots for **PRs** or **suggested fixes**, you define **branch protection**, **required reviewers**, and **scopes** so automation cannot bypass **intent**.
- **Incident response automation**: Patterns for **log clustering**, **runbook-driven** mitigation, **safe rollbacks** (image tags, feature flags)—**opt-in** destructive steps stay behind **explicit** approval.
- **Agent governance**: **Policies**, **RBAC**, **human-in-the-loop** checkpoints for production-adjacent actions; secrets never in plaintext in logs.
- **Monitoring & debugging**: Track **what** automation decided and **why** (inputs, outputs); watch for **repeated bad suggestions** or **drift** in prompts/policies.
- **Tooling integration (“skills”)**: Jira/Git/issue trackers, Git, **Terraform** / **Kubernetes** manifests when the stack needs them—**IaC** that agents and humans can both reason about.

### Familiar stacks (use what fits the repo)

**Agent frameworks** (LangChain, AutoGen, LangGraph, etc.), **LLMOps/observability** platforms (e.g. Weights & Biases–class tooling), **IaC**, and **prompt/policy iteration**—apply **pragmatically** per project; do not cargo-cult platforms the codebase does not use.
